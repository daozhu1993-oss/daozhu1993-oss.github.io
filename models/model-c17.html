焦糖布丁理论

Jobs-to-be-Done

用户雇的从来不是你的产品，而是墙上的那个洞

克莱顿·克里斯坦森  ·  Clayton Christensen  ·  《创新的两难》
← 返回思维模型图鉴 ← 回岛主主页
一句话总览： 克里斯坦森在《创新的两难》里提出 JTBD（Jobs To Be Done，用户任务理论）——用户不是买产品，是"雇用"产品来完成一个任务。就像买焦糖布丁，要的不是布丁本身，是它解决的那个场景：深夜独处时的一点甜，陪你熬过加班的夜。
Origin
克莱顿·克里斯坦森Clayton Christensen，《创新的两难》提出 JTBD 用户任务理论。
Field
产品创新 / 用户研究理解"用户为什么雇用你"的底层视角。
Core Definition
用户雇产品完成一个任务买的是"结果"，不是"工具"。
Thinking Layer
任务洞察层在功能清单之上，看清用户真正要干的那件事。

改功能 vs JTBD

Bad vs Good
✕

拼命改功能

Feature Obsession
  • 以为用户要"更好的钻头"
  • 功能越堆越多，用户却无动于衷
  • 盯着工具，忘了用户要的结果
  • 工具坏了他明天就换
例：以为用户要"更好的钻头"，其实他要的是"墙上的洞"，钻头再亮也不解决他的活。
✓

JTBD 视角

Hire for the Job
  • 先问用户雇产品去完成什么任务
  • 围绕"搬走限制"改产品，而非堆功能
  • 任务完成才被续雇，否则被解雇
  • 帮用户完成活，比功能多更重要
例：岛主买笔记软件，雇它"让脑子不漏事"，不是雇功能列表——漏事就解雇，界面再漂亮也没用。

三个关键拆解

Task · KPI · Constraint
01

任务（目标）

用户真正想完成什么。卖钻头不如卖"打洞"，卖吸尘器不如卖"十分钟干净的地板"，用户雇的是结果不是工具，工具坏了他明天就换。

02

期待（KPI）

用户对任务的具体指标。他心里有把隐形尺，达标才续雇你，不达标下次就换人，哪怕你功能再多，没帮他完成活就照解不误。

03

限制条件

完成中遇到的障碍。痛点才是你的入场券，谁帮用户搬走石头，谁被雇用，石头越大你越值钱，因为别人搬不动。

应用流程

Find the Job, Then Build

问任务

用户到底雇我的产品去完成什么任务？先把这个问清楚。

写 KPI

写出这个任务的指标和限制条件，把用户的隐形尺显性化。

搬走限制

围绕"搬走限制"改产品，而非堆功能——只留帮用户完成活的。

验证续雇

用户完成任务了吗？没完成就调整，直到他愿意续雇你。
