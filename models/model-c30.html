产品画布 MVP

MVP Canvas
埃里克·莱斯  ·  Eric Ries  ·  《精益创业》
一句话总览： MVP（最小可行产品）是「刚好够验证假设的最小闭环」——花 10% 的代价，先证伪最危险的假设，比 all in 后才知道错要体面得多，也更省眼泪。它不是糙产品，是聪明的试错。
Origin
埃里克·莱斯Eric Ries，《精益创业》提出 MVP——用最小代价验证商业可行性。
Field
产品 / 创业 / 验证凡是要「先测再建」、怕投入打水漂的场景都用得上。
Core Definition
最小可行产品只留一个核心功能 + 真能跑通一圈的最小闭环。
Thinking Layer
验证层 / 最小化试错在「写代码」之前，先确认「用户要不要」。

先做全再测 vs 先测最危险假设

Bad vs Good
✕

先做全再测

Build-It-All-First
  • 为点子闷头开发半年，上线却没人理
  • 投入全部身家，最后证明方向是错的
  • 代码写得漂亮，市场冷得结冰
  • 像精心装修了没人来的店，开业即关门
例：半年后才知道方向错，白白烧光时间和热情，连改的方向都没人给。
✓

先测最危险假设

Test the Riskiest First
  • 用最小代价，先证伪最危险的假设
  • 能手动就别写码，能假图就别真做
  • 验证了就加码，证伪了就 pivot
  • 用最糙的东西，问最真的问题
例：Dropbox 先放演示视频看有没有人想要，注册暴涨才写代码；Zappos 先拍照上架鞋、接到单再去进货，零库存验证「网上买鞋」。

MVP 的几个关键点

The MVP Keys
01

最小——只留一个核心功能

凡是不验证「用户要不要」的功能都是浪费，多做一天多烧一天。早一天验证早一天止损，少即是快。

02

可行——真能跑通一圈

demo 再漂亮跑不通闭环就是摆设，用户要的是完整体验不是截图，断一环就露馅，闭环才算数。

03

卖点 / 关系 / 整体性

画布要回答：凭什么买、和谁发生关系、是不是一个完整价值。缺了共鸣性用户记不住你，情感连接才是留存。

04

MVP 不是糙产品

它是「刚好够验证假设的最小闭环」。花 10% 的代价先证伪最危险的假设，比 all in 后才知道错要体面得多。

岛主经历：想做社群工具没先写码，拉个微信群手动模拟，三天发现用户根本不想每天打卡——省下三月开发。
05

敌人是完美主义

能手动就别写码，能假图就别真做。判断 MVP 成不成功，不看它多精致，看「假设被验证了没有」。

06

只为最危险假设而生

多一个功能就多一层噪音，也多一周白等。最怕验证出「用户不要」还硬说「再改改就好」。

三步做 MVP

Build Your MVP

写下最赌的假设

写下你最赌的那个假设（如「用户愿为 X 付费」），先想清楚证伪条件。

最低成本做小东西

用最低成本做出能验证它的小东西，哪怕手动、哪怕丑。

看真实行为

看真实行为，不是嘴上说「挺好」，再决定加不加码，别被客气话骗了。

证伪就 pivot

验证了就加码，证伪了就 pivot，别硬说「再改改就好」——那是在骗自己。

模型金句

In a Nutshell

MVP 的本质

聪明人用 MVP 试错，笨人用全部家当试错。

试错的代价
